# Feature — Fun Facts (AI) — Detailed

Spanish-language trivia paragraphs about albums and songs, generated by Gemini or Groq and cached permanently. Displayed in the now-playing card when the admin sets an album.

## User experience

When the admin sets now-playing (album scope), the now-playing card includes a "fun facts" section. On load:
1. Client calls `GET /api/fun-facts?album=...&artist=...`.
2. If 15+ facts are cached: returned immediately.
3. If fewer than 15: server fetches inline (blocking the request until the LLM responds or times out).
4. Facts display as a rotating card or scrollable list in the UI.

## Prompt design

Both Gemini and Groq receive the same prompt (in Spanish), asking for:
- 15 facts about the album
- Mix of: album history/recording context, individual song facts, music video facts
- Written in Spanish, with artist/album/song/award names staying in English
- Genuinely interesting (not generic)

Gemini receives the prompt as a raw text generation request. Groq receives it wrapped in a chat message with `response_format: { type: "json_object" }`, which forces structured JSON output — making parsing more reliable.

The `existing_facts` parameter is passed when the cache already has some facts (< 15). Both prompts include a "don't repeat these" instruction to avoid duplicate facts when the background worker tries to fill in the remaining slots.

## Gemini implementation (`_call_gemini`)

- Model: `gemini-2.0-flash-lite`
- Endpoint: `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash-lite:generateContent?key=...`
- `maxOutputTokens: 1800`, `temperature: 0.75`
- Rate limiting: `_gemini_rate_lock` enforces `_GEMINI_MIN_INTERVAL = 5.0` seconds between calls. This is enforced globally, not per-thread.
- Response parsing: extracts text from `candidates[0].content.parts[0].text`, strips markdown code fences if present, parses as JSON array.
- 429 handling: parses `RetryInfo` from the error response body to extract the exact `retryDelay`. Sets `_gemini_last_call = time.time() + retry_s` (with +5s buffer) so subsequent calls block for the right duration. Returns `None` as a sentinel to trigger Groq fallback.

## Groq implementation (`_call_groq`)

- Model: `llama-3.1-8b-instant` (note: README says `llama-3.3-70b-versatile` but the actual code uses `llama-3.1-8b-instant`)
- Endpoint: `https://api.groq.com/openai/v1/chat/completions`
- `max_tokens: 1800`, `temperature: 0.5`
- `response_format: { type: "json_object" }` — forces JSON mode, avoiding markdown wrapping issues.
- Response parsing: parses the JSON, extracts `parsed["facts"]` or the entire payload if `facts` key is absent.

## Fallback logic (`_fetch_fun_facts`)

```python
result = _call_gemini(album, artist, existing)
if result is None:      # Gemini returned the 429 sentinel
    result = _call_groq(album, artist, existing)
return result or []
```

If Gemini returns an empty list (not None): Groq is NOT tried. This distinguishes "Gemini failed outright" from "Gemini is rate-limited and Groq should be tried now."

## Background prefetch worker

See [[fun-facts]] for the full prefetch flow. Key detail: the worker does not retry a failed album immediately — it moves to the next one and sleeps 5 seconds after any failure. This means an album that consistently fails (e.g. very obscure, no response from either provider) won't block the entire queue indefinitely.

## In-memory cache (`_fun_facts_db`)

Loaded from disk at startup into `_fun_facts_db` dict. All reads and writes go through `_fun_facts_lock`. The in-memory copy is the source of truth for serving requests; the file is a persistence layer.

Cache key format: `"album title lowercase::artist name lowercase"`. Example: `"abbey road::the beatles"`.

## Graceful degradation

- No `GEMINI_API_KEY`: Gemini is skipped entirely.
- No `GROQ_API_KEY`: Groq is skipped entirely.
- Both missing: `GET /api/fun-facts` returns `{ "facts": [] }`.
- The server starts and operates normally — fun facts are purely optional.

## Disk persistence

`_write_fun_facts_db(db)` writes the full dict to `fun-facts-db.json` on every successful fetch. If the file write fails (disk full, permissions), the error is silently swallowed — the in-memory cache is unaffected and facts remain available for the session.
